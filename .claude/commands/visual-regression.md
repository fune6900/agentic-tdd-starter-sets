Playwright MCPを使用して視覚的整合性の自動検閲を行う。図案のメイド（Designerサブエージェント）を起動し、以下の手順を厳守すること。

## 対象
- 引数 $ARGUMENTS に検査対象のURLまたはページパスを指定する。
- 未指定の場合は `http://localhost:3000` を起点に主要ページを巡回する。
- カンマ区切りで複数URL指定可能（例: `http://localhost:3000,http://localhost:3000/search`）。

## 手順

### Phase 1: ベースライン確認（デスクトップ）
- `mcp__playwright__browser_navigate` で対象URLへ遷移する。
- `mcp__playwright__browser_take_screenshot` でデスクトップ（1280x800）のスクリーンショットを取得する。
- `mcp__playwright__browser_snapshot` でアクセシビリティツリーとDOM構造を取得する。

### Phase 2: インタラクティブ・ステート検証
各ページで以下の状態を順に操作し、その瞬間のスクリーンショットを撮影する。

1. **通常状態 (Default)**: ページ初期表示をキャプチャ。
2. **ホバー状態 (Hover)**: `mcp__playwright__browser_hover` で主要ボタン・リンクにホバーしてキャプチャ。
3. **フォーカス状態 (Focus)**: `mcp__playwright__browser_press_key` (Tab) でフォーカスを移動しキャプチャ。
4. **入力状態 (Input)**: フォームが存在する場合 `mcp__playwright__browser_type` でテキスト入力しキャプチャ。
5. **エラー状態 (Error)**: 不正な値を入力し送信、エラーメッセージ表示をキャプチャ。
6. **ローディング状態 (Loading)**: `mcp__playwright__browser_evaluate` でネットワーク遅延を模倣し、スピナー等の表示をキャプチャ。

### Phase 3: レスポンシブ・ストレステスト
`mcp__playwright__browser_resize` で以下のビューポートに順次リサイズしてスクリーンショットを取得する。

| デバイス | 幅 | 高さ |
|---------|-----|------|
| iPhone SE | 375 | 667 |
| iPhone 14 Pro | 393 | 852 |
| iPad | 768 | 1024 |
| Laptop | 1280 | 800 |
| Full HD | 1920 | 1080 |

各解像度で以下を確認する:
- テキストが溢れていないか（overflow: hidden でカットされていないか）
- ボタン・リンクが重なっていないか
- ナビゲーションがモバイルで正常に折りたたまれているか
- 画像のアスペクト比が崩れていないか

### Phase 4: UX動線追跡
主要フロー（以下を参考に、ページの性質に応じて実行）を実際に操作し、各ステップの応答時間を計測する。

1. **ナレッジ検索フロー**: 検索フォームにキーワード入力 → 送信 → 結果表示までの時間
2. **AI試着フロー**: 商品選択 → 試着ボタン → ローディング → 結果表示までの時間
3. **コーデ日記フロー**: 投稿ボタン → フォーム表示 → 入力 → 保存 → 完了表示までの時間

`mcp__playwright__browser_evaluate` で `performance.now()` を使用し、操作間隔（ms）を計測する。

### Phase 5: コンソールエラー確認
- `mcp__playwright__browser_console_messages` でコンソール出力を取得する。
- エラー（error）・警告（warning）を抽出し重大度を判定する。

## 報告形式

以下の形式で検閲結果を報告すること。感傷は不要。事実のみ。

```
## 視覚的整合性 検閲報告: <対象URL>

### 総合判定: [合格 / 要修正 / 緊急対応]

---

### Phase 1: デスクトップ表示
- レイアウト: [正常 / 異常の詳細]
- フォント・色の一貫性: [正常 / 差異の詳細（px単位まで）]

### Phase 2: インタラクティブ・ステート
| 状態 | 判定 | 詳細 |
|------|------|------|
| Default | OK / NG | 所見 |
| Hover | OK / NG | 所見（影のサイズ差、色変化の差分等） |
| Focus | OK / NG | 所見（フォーカスリングの視認性等） |
| Input | OK / NG | 所見 |
| Error | OK / NG | 所見（エラーメッセージ表示遅延: XXms等） |
| Loading | OK / NG | 所見（スピナー位置ズレ: Xpx等） |

### Phase 3: レスポンシブ検証
| デバイス | 判定 | 検出された問題 |
|---------|------|---------------|
| iPhone SE (375px) | OK / NG | 詳細 |
| iPhone 14 Pro (393px) | OK / NG | 詳細 |
| iPad (768px) | OK / NG | 詳細 |
| Laptop (1280px) | OK / NG | 詳細 |
| Full HD (1920px) | OK / NG | 詳細 |

### Phase 4: UX動線計測
| フロー | ステップ数 | 最長待機 | 判定 |
|--------|-----------|---------|------|
| ナレッジ検索 | X steps | XXXms | OK / 要改善 |
| AI試着 | X steps | XXXms | OK / 要改善 |
| コーデ日記 | X steps | XXXms | OK / 要改善 |

### Phase 5: コンソール出力
- エラー: X件（詳細）
- 警告: X件（詳細）

---

### 検閲指摘事項（修正命令）
1. [緊急] 具体的な問題と修正箇所（コンポーネント名:行数）
2. [要対応] 具体的な問題と修正箇所
3. [推奨] 具体的な問題と修正箇所

### 修正方針
（コード変更が必要な場合、対象ファイルと具体的な修正内容を記載）
```

## 注意
- 開発サーバーが起動していない場合は `npm run dev` の実行をマスターに促すこと。
- 「なんとなく崩れている」という曖昧な報告は不可。ズレはpx単位、遅延はms単位で報告すること。
- `mcp__playwright__browser_evaluate` でCSSの計算値（`getComputedStyle`）を取得し、設計値との乖離を数値で示すこと。
- スクリーンショットで視認できた問題は、DOM inspectionで根本原因（どのCSSプロパティが原因か）まで特定すること。
- 修正提案は「色を調整しましょう」のような曖昧なものは不可。`className` の変更、`style` プロパティの具体値まで落とし込むこと。
- 複数URLが指定された場合は、全URLに対してPhase 1〜5を実行してから総合報告すること。
